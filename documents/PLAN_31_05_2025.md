Задача 1: Итеративная адаптация Front-end к изменениям в API (связанным с новой схемой БД) – БЕЗ ИЗМЕНЕНИЯ ДИЗАЙНА


Описание: Начать процесс адаптации Front-end кода к изменениям в API бэкенда, которые возникли вследствие перехода на новую схему базы данных. Поскольку полный объем изменений в API и их влияние на Front-end заранее неизвестны, эта задача будет выполняться итеративно. Критически важно: все изменения должны касаться только логики получения, обработки и отображения данных; визуальный дизайн компонентов НЕ должен изменяться.

Требования:
Подготовительный этап:
Ознакомиться с любыми доступными описаниями изменений в API (если есть) или быть готовым анализировать ответы API в процессе.
Проверить тестовую среду, убедиться, что она использует обновленный бэкенд.
Итеративный процесс:
Каждая итерация должна быть сфокусирована на минимальном, атомарном изменении. Например, адаптация одного компонента, который запрашивает или отображает определенные данные, или обновление одного API-вызова.
Идентифицировать затронутые компоненты/сервисы: Определить, какие части Front-end кода (компоненты, сервисы API-вызовов, хранилища состояния типа Redux/Zustand) затронуты предполагаемыми изменениями в структуре данных или эндпоинтах API.
Внесение изменений (только логика):
Аккуратно внести необходимые изменения в код Front-end для корректного взаимодействия с обновленным API. Это может включать:
Изменение URL эндпоинтов, если они были переименованы.
Корректировка структуры отправляемых запросов (payload).
Адаптация обработки ответов API (парсинг новой структуры JSON).
Изменение моделей данных на Front-end или интерфейсов TypeScript.
Обновление логики маппинга данных на поля компонентов.
ВАЖНО: Если для корректного отображения новых данных требуется изменение дизайна (например, нужно отобразить новое поле, для которого нет места в текущем UI), Остановись и спроси пользователя!!!
Учти, что теперь лоигка бэкэнда строится на датах, а не weekid и дне недели. теперь фронтэнд сам формирует и вычилсяет половины недели для отрисовки, а отправляет и принимает конкретные даты.
Задокументировать проблему и сообщить об этом для принятия решения.
Проверка и тестирование (ключевой шаг каждой итерации):
После каждого значимого изменения в коде Front-end, выполнить тестирование:
Запустить существующие автоматизированные тесты (юнит-тесты компонентов, e2e тесты, если есть).
Испрвь тесты, если это необходимо.
Ручное тестирование функционала, затронутого изменениями. Убедиться, что:
Данные корректно запрашиваются у бэкенда.
Данные корректно обрабатываются и отображаются в UI.
Визуальный дизайн остался неизменным. Сравнить с предыдущей рабочей версией или скриншотами.
Отсутствуют ошибки в консоли браузера.
Фиксация: После успешной проверки и тестирования, зафиксировать изменения в системе контроля версий.

Важные принципы:
Принцип "Одна вещь за раз": Не изменять много компонентов или API-вызовов одновременно.
Постоянные тесты: Тестирование — неотъемлемая часть каждой итерации.
Не изменять дизайн: Любые необходимые дизайн-изменения должны быть вынесены в отдельные задачи после явного согласования. Приоритет — функциональная адаптация.
Обратная связь: В случае блокировок (например, невозможность отобразить данные без изменения дизайна) или неясностей в работе API, немедленно сообщать.

Критерии приемки (для общего завершения задачи, не для каждой итерации):
Front-end код полностью адаптирован для работы с обновленным API бэкенда.
Все компоненты и страницы, использующие данные из API, корректно получают, обрабатывают и отображают эти данные.
Визуальный дизайн и пользовательский интерфейс остались неизменными по сравнению с версией до начала адаптации.
Отсутствуют ошибки в консоли браузера, связанные с получением или обработкой данных.
Все автоматизированные тесты, если они есть и релевантны, проходят успешно.
